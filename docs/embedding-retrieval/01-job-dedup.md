### Abstract (초록)

> 여러 온라인 채널에 게시되는 채용공고의 중복을 탐지하는 방법을 제안
> 
- 동일한 채용 건이 플랫폼마다 요구사항으로 인해 다른 텍스트로 게시됨
- 문자열 유사도, 텍스트 임베딩, 가중치 키워드 매칭을 결합한 접근법 제시
- 실제 프로덕션 환경에서 사용 중이며 긍정적인 피드백 확인

### 1. Introduction (서론)

- 여러 소스에서 수집된 데이터 병합 시 중복 탐지가 필수적
- 단순하고 자동화된 중복 검사는 대부분 높은 오탐률 → 수동 작업의 형태로 해결되는 경우가 대부분
- 가장 좋은 결과를 냈던 `접근 방법`  : 문자열 유사도 + 도메인 지식(가중치 스킬) + 텍스트 임베딩

### 2. Industrial Use Case and Requirements (산업 사례 및 요구사항)

#### 프로젝트 제약 조건

- 수동 중복 탐지 인건비보다 저렴해야하 함(15~20인 개발)
- 라이선스 비용 및 하드웨어 렌탈 비용 없어야함
- 최종 사용자가 유지보수 가능한 단순한 솔루션 선호

### 3. Related Work (관련 연구)

#### 주요 선행 연구

- 채용 분야 AI : 주로 이력서/지원서 분석에 집중
- 중복 탐지 : Jaccard 유사도, n-gram 등 전통적 기법의 한계 확인
- 임베딩 활용: Word2Vec 등을 활용한 단문 중복 탐지 (채용공고는 긴 문서라 다른 접근 필요)
- 텍스트 유사도: Levenshtein 거리, TF-IDF, LCS 등은 채용공고에 부적합 (유사 단어/문자 많지만 의미는 다름)
- 표절 탐지 : 관련은 있지만 동일하진 않음, 새로운 머신러닝 기법을 조합하여 사용하는것이 유망하다는 내용
- 텍스트 유사도 측정은 응용 분야와 도메인에 특화적 → 수행해야 할 작업에 따라 달라짐

### 4. Data Resources (데이터 리소스)

- 45,000개 데이터
- title(공고 제목), description(본문, 회사 소개, 업무 내용, 요구사항, 근무 조건 등)
    - 게시 날짜와 같은 일부 메타 정보도 포함됨
    - 근무지 , 기간
- 기술 목록 (이력서에서 최소 7번 출현한 기술 - 25,000개)
- 기술 블랙리스트 (기술 목록에서 오해의 소지가 있는 항목 제외용 )
- 테스트 세트 : 50개 중복 쌍(수동으로 주석 생성)
- 검증 세트 : 176개 (74개 중복 + 73개 비중복)
    - 비중복인 73개는 알고리즘이 중복으로 판단했으니 실제로는 아닌 것
    - 29개는 도메인 전문가의 검토를 거쳤음에도 중복/비중복 판단이 어려워서 제거

### 5. Method and Experimental Setting (방법론 및 실험 설정)

#### 5.1 Data Preprocessing (데이터 전처리)

`정규화` 

- 특수 문자(줄바꿈, 번호 등 열거형) 제거
    - 기술 스킬에 나타나는 특수 문자는 유지 ex) C++, S/4 Hana
- 문자, 공백, 숫자 정규화
- 4.Data Resources에서 언급한 기술 목록과 기술 블랙리스트를 가중치와 함께 원본 설명 텍스트에서 모든 기술 용어를 추출하고 소문자로 변환

#### 5.2 Evaluation Metrics (평가 지표)

`지표`

- 정밀도, 재현률 ,F1-score → 표준 지표 사용
- TP,FN,FP,TN 계산

`임계값Threshold)` 

- 점수 의존적 임계값으로 중복/비중복 라벨 할당
    - 일치하는 쌍 샘플에 알고리즘에 의해 임계값보다 높으면 → 중복 레이블 할당
    - 임계값보다 낮으면 → 비중복 레이블 할당

#### 5.3 Overlap-based Similarity Matching (유사도 매칭)

`TOS(Text Overlap Score)`

- TOS = B / len(S) → 겹치는 문자열 블록의 길이
    - 결과를 정규화하여 겹침 정도를 나타내는 백분율 값을 얻는다
- B : 겹치는 텍스트 블록의 합(최소 15자 이상)
    
    $$
    B = \sum_{i=1}^{n} b_{i}
    $$
    
    - S를 원본 텍스트로 정의하고 T를 대상 일치 텍스트로 정의한다. 각 중첩 블록 i에 대해 bi를 그 길이로 정의한다. 총 n개의 중첩 블록이 존재할 경우, 중첩 텍스트의 총 길이는
- len(S) : 원본 텍스트 길이
    
    **`첫 번째 테스트 결과`** 
    
    - 회사 소개나 연락처 같은 공통 상용구 때문에 실제 내용이 다른 공고들도 점수가 높게 나오는 문제
        
        → 공통된 상용구 때문에 텍스트 중복 점수만으로는 구별하기 어렵다. (SOS로 보완)
        
    - 공고 간의 차이를 구별하는 데 **가장 중요한 부분은 '프로젝트 관련 기술 요구사항' 섹션**
        
        → 공고를 구별하는 진짜 알맹이는 기술 요구사항 부분임을 발견
        

`SOS(Skil Overlap Score)`

- 전체 텍스트가 아닌 추출된 기술 키워드들만을 대상으로 중복 점수를 계산
- 앞서 기술 요구사항 섹션이 공고 간의 차이를 구별하는 데 가장 중요함을 발견했으므로 기술 요구사항에만 집중함으로써 상용구로 인한 오류를 줄이고 정확도를 높임

`최종 점수 계산`

- TOS,SOS 모두 A를 기준으로 B를 비교한 점수와 
B를 기준으로 A를 비교한 점수의 **평균 값**을 최종 점수로 사용
    
    → 비교 순서에 따른 오차를 줄이기 위함
    

$$
TOS_{\text{final}} = \frac{TOS_{\text{forward}} + TOS_{\text{backward}}}{2}
$$

#### 5.4 Embedding-based Similarity Matching (유사도 매칭)

> 문자열 중복 점수로 감지되지 않는 의미론적 유사성(다른 단어나 구문)을 포착하기 위해 텍스트 임베딩에 기반한 유사도 매칭 테스트 진행
> 
- 독일어, 영어 채용 공고가 포함되어 있어서 sentence-transformers에서 제공하는 distiluse-base-multilingual-cased-v1 모델을 사용하여 다국어 텍스트 인코딩 방법을 사용

`계산 방법` 

- **텍스트 임베딩 점수 (TES):** 전체 채용 공고 텍스트에 대한 임베딩
- **기술 임베딩 점수 (SES):** 기술 키워드만 모은 텍스트에 대한 임베딩
- **제목 임베딩 점수 (TTES):** 채용 공고의 title에 대한 임베딩
- **전체 텍스트 임베딩 점수 (AES):** 모든 텍스트에 대한 임베딩

#### 5.5 Weighted Keyword Matching (가중치 키워드 매칭)

- 5.3에서 채용 공고에 언급된 기술 용어가 중복 탐지 작업과 관련하여 핵심 특징임을 발견함
- 반명, 관련 기술 목록이 자동화된 방식으로 생성되었기 때문에 “영어”, “경험”, “팀워크”와 같은 잘못되거나 오해의 소지가 있는 키워드도 포함하고 있음
    - “JAVA”와 같이 많은 채용공고에 등장하는 흔한 용어는 중복을 찾는데 도움도 안됨

→ 가중치 기술 점수(WSS)를 도입 

**`WSS** (Weighted Skill Score)`

$$
w(s_{i}) = \frac{1}{f(s_{i})}
$$

- w(s_i) : 가중치
- F(s_i) : 전체 데이터베이스에서 스킬 s_i의 빈도
    - 희귀한 스킬일수록 높은 가중치
    - 자주 등장하는 단어 ex) JAVA → 낮은 가중치

---

$$
WSS = \frac{\sum_{s_{i} \in (S \cap T)} w(s_{i})}{\sum_{s_{i} \in S} w(s_{i})}
$$

- WSS는 두 직무 기술서 S와 T 모두에서 발견되는 모든 기술 용어의 가중치 합계를 매칭 쌍의 소스 텍스트 내 모든 기술의 가중치 합계로 나눈 값

→ 두 공고에서 공통으로 발견되는 기술들의 가중치 합을 원본 공고에 있는 모든 기술의 가중치 합으로 나눔 

- 희귀하고 핵심적인 기술이 일치할 때 더 높은 점수를 부여하여 흔한 기술 몇 개가 겹치는 것보다 더 의미 있는 유사성을 측정할 수 있음

---

$$
WSS_{\text{final}} = \frac{WSS_{\text{forward}} + WSS_{\text{backward}}}{2}
$$

- 다시 S와 T를 서로 바꾸어 정방향, 및 역방향 계산을 수행한 값의 평균을 WSS로 사용(앞에서 했던 TOS 최종 계산 방법과 동일함)

### 6 Results and Discussion (결과 및 논의)

#### 6.1 Threshold Finding (임계값 설정)

- 프로젝트의 제한된 범위로 인해 임계값은 사람이 수동으로 설정
- 비교 가능성을 위해 모든 그룹에 동일한 값을 사용했음

#### 6.2 Experimental Results (실험 결과)

- **전체 텍스트 기반 점수 (TOS, TES)의 한계:** 상용구(boilerplate) 텍스트 때문에 성능이 저조했음. 같은 회사에서 올린 다른 공고를 중복으로 오인하는 경우가 많았기 때문
- **기술 텍스트 기반 점수 (SOS, SES)의 우수성:** 핵심인 '기술' 부분에 집중하자 성능이 크게 향상됨
- **임베딩의 한계:** 임베딩 점수들은 의미적으로 유사한 '가짜 중복'과 '진짜 중복'을 잘 구분하지 못하는 경향을 보임
- **가중치 기술 점수 (WSS)의 강력함:** 중요한 기술에 가중치를 부여하는 WSS는 단독으로도 매우 높은 성능을 보임→ 상용구의 영향을 효과적으로 줄였기 때문

### 7 Productive Deployment(실제 적용)

- 배포된 시스템의 경우 점수 계산을 위한 임계값이 조정되었고 추가 기준을 도입했음
- TS 임계값은 0.6으로 설정되고 각 개별 점수 방법은 중복으로 분류하기 위해 최소 0.1을 달성해야함
    
    → 이 설정으로 긍정적인 피드백을 받았다…
    

### 8. Limitations(한계점)

- 샘플 테스트가 작을 때 설정한 임계값은 데이터 세트가 커졌을 때 적절하지 않을 수 있음
    
    → 일반화 가능한 임계값을 추정해야함 
    
- 더 개선하려면 총점인 TS를 형성하는 개별 지표에 할당된 가중치를 더 정교하게 결정하여 개선할 수 있음

나머지 노드들도 통합할 수 있거나 따로 유틸 파일로 빠질 수 있거나 등 여러가지 방법들을 검토해서 노드들을 최대한 간소화 할 수 있는 방안을 알려줘 plan이나 관련 문서를 작성할때 한국어로 작성해줘
