## 초록

- 고객 서비스 챗봇과 같은 애플리케이션 같은 경우 빈번한 API 호출로 높은 비용과 병목 현상이 나타남
- 사용자 쿼리를 임베딩 함으로써 유사한 질문을 효율적으로 사용 → 중복 API 호출 줄이기 가능

#### 결과 미리보기

- 다양한 쿼리 범주에서 최대 68.8% 중복 호출을 줄임
- 높은 정확도를 달성하여 긍정적 적중률이 97% 초과
- 비용 절감 뿐만 아니라 응답 시간을 개선함

---

## 서론

- 각 사용자 쿼리에 대해 LLM이 개별 API를 호출해야 한다는 문제 발생
    - 특히 고객 서비스 시나리오에서 유사하거나 반복적인 질문
    - 대량으로 처리할경우
    
    → 중복 처리 및 높은 계산 비용 발생
    
- 쿼리 임베딩의 시맨틱 캐싱을 통해 지연 시간 및 비용 문제를 해결
    
    →의미상 유사한 쿼리에 대한 응답을 효율적으로 검색할 수 있도록 함으로써 보안
    

#### 장점

- 지연 시간 감소
    - 시스템이 캐시에서 직접 응답 검색 → 사용자는 더 빠른 응답을 받음
- 비용 효율성
    - 중복 API 호출이 줄어 운영 비용 절감
- 확장성
    - 계산 리소스가 비례적으로 증가하지 않음 → 더 많은 양의 쿼리 처리 가능
- Local Model

## 시스템 아키텍처

### 개요 (아키텍처 구성 요소)

- Embedding Generation
    - 사용자 쿼리를 의미적 임베딩으로 변환
- In-Memory Caching
    - Redis를 사용해서 임베딩 및 응답의 저장 및 검색 관리
- Similarity Search: Approximate Nearest Neighbor (ANN)
    - ANN을 사용하여 의미상 유사한 쿼리를 식별

#### Embedding Generation

- OpenAI API 또는 허킹페이스의 ONNX 호환 모델 활용 가능
    
    → 자체 호스팅 옵션, 다양한 배포 시나리오에 적용 가능
    

**임베딩 생성 과정** 

OpenAI 

- 텍스트가 API로 전송되고 응답에 임베딩 벡터 포함
- 벡터 길이 : text-embedding-ada-002 모델의 경우 1536

Local Models 

- ONNX 지원 모델을 로드하고 @xenova/transformers와 같은 라이브러리를 사용
- 생성된 임베딩은 정규화되고 풀링되어 일관된 표현 생성

### In-Memory Caching

- 캐싱 레이어는 Redis를 사용

**Embedding Size**

- 임베딩 모델에 따라 다양한 차원의 벡터가 생성 → 캐시는 임베딩 크기를 기준으로 분할
- TTL(Time-To-Live) : 캐시된 데이터는 TTL이 할당 → 오래된 정보 무기한 유지 X, 자동 만료

### ANN을 사용한 유사성 검색

- ANN 알고리즘을 사용하여 효율적인 유사성 검색 수행
- 코사인 유사도를 사용하여 k개의 가장 유사한 임베딩을 검색함
    - HNSW 그래프의 다층 순회 포함
    - 진입점에서 시작 → 상위 k개의 가장 가까운 노드로 수렴할때까지 유사도 계산 → 가장 가까운 이웃 탐색

### 쿼리 처리 워크플로우

1. 캐시 조회
    - 쿼리 수신 → 임베딩 → ANN 통해 유사한 임베딩 검색 → 0.8 이상 캐시된 응답이 반환
2. 캐시 미스 
    - 적합한 항목이 없으면 LLM API 호출 → 새 응답 생성 → 캐시 저장(나중에 사용하기 위함)
3. 임베딩 저장 
    - 새로운 임베딩과 응답이 ANN index와 Redis 캐시에 추가
    - 시트템이 캐시 크기를 관리하고 필요할 때 크기를 조정하여 인덱스가 증가하는 데이터를 수용할 수 있도록 관리

### 코사인 유사도 및 유사도 임계값

- 쿼리 임베딩 간의 유사성을 측정하는 데 사용
- -1 ~ 1 사이 값 (1은 동일)
    - 높은 유사도 → 더 큰 시맨틱 접근성 → 얼마나 동일한가?

### TTL을 사용한 캐시 관리

- 캐시가 오래되는 것을 방지 → 캐시 크기 방지
- 특정 기간 후에 데이터가 자동으로 제거

### 시스템 워크플로우 요약

1. 사용자가 쿼리 제출
2. 임베딩 생성 (API 또는 로컬 모델0
3. ANN 통해 유사한 임베딩 검색
4. 코사인 유사도 계산
5. 임계값 이상으로 발견된 캐시 응답 반환
    - 혹은 발견되지 않으면 LLM API로 새로운 응답 생성 → 새로운 응답 캐싱
6. 캐시 및 ANN index 업데이트 

### 아키텍처의 장점

- 지연 시간 감소
- 비용 절감
- 확장성
    - ANN과 인-메모리를 사용하여 증가하는 쿼리 양을 적절히 처리 가능
- 유연성
    - 여러 임베딩 모델 사용 가능

### 확장 가능성

- 동적으로 임계값 조정
- 분산 캐싱으로 확장
- 고급 임베딩 모델 사용 →의미론족 이해 개선 또는 최신 도메일별 임베딩 모델 통합

## 테스트 방법론

#### 데이터 세트 준비 및 캐시 준비

- Python 프로그래밍, 고객 서비스 문의, 기술 지원 질문 및 일반 지식의 네 가지 범주에 걸쳐 8,000개의 질문-답변 데이터 세트 구성
- 질문은 all-MiniLM-L6-v2 (Reimers & Gurevych, 2020)를 사용하여 의미론적 임베딩으로 변환

#### 테스트 쿼리 생성 및 실행

- 범주별로 500개 쿼리를 포함한 추가 2000개 테스트 쿼리
- 유사성 0.8 이상 → 캐시 응답
    - 그렇지 않으면 LLM 으로 전달 후 이후 캐싱

#### 캐시 적중 유효성 검사

- GPT-4o Mini 사용
- 캐시 적중에 대해 테스트 쿼리와 원래 캐시된 질문이 모두 모델에 제공
    - 쿼리가 의미상 유사한가? 1 or 0
    - 캐시된 응답이 정환한가? 1 or 0
    
    → 2진  평가 진행

## 결과

#### API 호출 감소

- Python 프로그래밍 기초: 67% 캐시 적중률, API 호출 33% 감소
- 네트워크 관련 기술 지원: 67% 캐시 적중률, API 호출 33% 감소
- 주문 및 배송 관련 질문: 68.8% 캐시 적중률, API 호출 31.2% 감소
- 고객 쇼핑 QA: 61.6% 캐시 적중률, API 호출 38.4% 감소

#### 응답 시간 감소

#### 캐시 히트 및 정확도

- 모든 범주에서 92.5% 달성
- positive hit rate 97.3%로 신뢰성 확보

## Discussion

- 챗봇과 같은 애플리케이션에 매우 적합
- 운영 비용 절감 효과가 상당함
- 구조화되거나 반복적인 쿼리가 있는 범주에서는 성능이 좋았음
    - 소비자 쇼핑 QA같은 다양한 쿼리 유형에서는 제한 사항이 관찰됨
    - 광범위한 다양성과 사용된 언어의 의미적 가변성으로 적중률이 61.6%로 낮았음
    - 임계값(0.8)을 사례에 맞게 커스텀 해야 높은 적중률 가능할 것
- 대규모 데이터 세트에 대한 ANN 인덱스 유지가 힘듦
    - 실시간 사용자 입력 스트림을 받는 경우 → 대기시간 및 일관성 문제

#### 임계값을 0.8로 설정한 이유

- 임계값이 0.8 미만이면 캐시 적중률이 높아졌지만 관련 없는 일치 항목도 도입되어 긍정적 적중률이 감소하고 결과적으로 전체 응답 정확도가 감소했음
- 임계값이 0.8을 초과하면 의미상 유사한 쿼리로 식별되는 쿼리가 줄어들어 캐시 적중률이 크게 감소하여 더 많은 LLM API 호출과 더 높은 비용이 발생
- 0.8에서 시스템은 최적의 균형이라고 판단함 → 응답 정확도를 저하시키지 않고(긍정적 적중률이 97% 초과) 높은 캐시 적중률(최대 68.8%)을 유지

#### 주요 사용 사례

- 고객 지원 챗봇
- RAG 시스템
- 전자 상거래 제품 지원
