## 초록

> +++Reasoning, +++Tone(style=formal), +++Import(topic="Systems Thinking")와 같은 간결한 제어 토큰을 통해 LLM의 동작을 제어하는 선언적이고 조합 가능한 문법인 프롬프트 데코레이터를 소개
> 
- 각 데코레이터는 작업 내용을 변경하지 않고 추론 스타일, 구조 또는 톤과 같은 동작을 수정가능
- 인지 및 생성/표현 민 시스템으로 구성됨
    - 각 계열은 추론, 상호작용, 표현 및 세션 제어를 관리하는 하위 범주로 더 분해
    
    → 작업 의도를 실행 동작에서 분리함으로써 재사용 가능하고 해석 가능한 인터페이스를 생성함 
    
    ex) 추론 투명성 향상, 프롬프트 복잡성 감소, 도메인에 상관없이 표준화된 모델 동작 
    

## 1. 서론

- GTP, Claude, Gemini, LLama와 같은 LLM이 어떻게 추론/작동하고 제어하는 문제는 여전히 남아있음
- 현재의 프롬프트 관행 :
    - LLM의 행동을 제어하기 위해 매번 다른 방식으로 긴 설명을 작성하고 실험을 해야함
        - 추론 : 단계별로 생각해서 답변해줘
        - 어조 : 공식적인 어조로 답변해줘
        - 구조 : 마크다운 형식으로 답변해줘
    
    자연어에 의존적임 → 직관적이지만 표준화, 재현성, 투명성이 부족함 
    ⇒ 예측 불가능 및 일관성 없는 결과 
    
    - 자연어는 표현력이 풍부하지만, `제어` 하는데는 신뢰할 수 없음
    - 작은 문법이나 어휘만 바뀌어도 추론 및 출력 스타일이 큰 동작 편차를 유발할 수 있기 때문

### Prompt Decorators

- 프롬프트의 의미적 내용을 변경하지 않고, 모델이 **어떻게** 추론하고, 형식을 지정하고, 상호작용할지를 지정하는 선언적이고 조합 가능한 구문
- 각 데코레이터는 구조화되었지만 사람이 읽을 수 있는 형태의 간결한 행동 지시를 나타냄
    
    프롬프트 예시 : 
    
    ```
    +++Reasoning
    +++Debate
    
    Explain the implications of using facial recognition in public spaces.
    (공공장소에서 안면 인식 기술 사용의 함의를 설명하시오)
    ```
    
    1. 결론을 제공하기 전에 추론을 명시적으로 설명하고 `+++Reasoning` 
    2. 출력을 마크다운 형식으로 지정하도록 지시 `++Debate` 
    
    → 자연어 지시 대신 선언적 기호를 통해 간결하게 표현
    

#### Prompt Decorators의 핵심 기능

1. 선언성 : 사용자가 원하는 추론 및 출력 행동을 명시적으로 정의하고, 자연어 표현과 분리한다
2. 조합성 : 여러 데코레이터를 모듈형식으로 만들어 조합하여 사용하고, 재사용 가능하다
3. 투명성 : 행동 논리를 명시적으로 노출하여 재현성과 해석 가능성을 향상시킨다

#### 프레임워크 설명

- 인지 및 생성, 표현 및 시스템으로 그룹화된 20개의 핵심 데코레이터를 형식화함
- LLM에게 지시한 작업과 행동 지시를 분리함으로써 프롬프트 설계를 구조화하고 감사 가능하게한다

## 2. 관련 연구 및 배경

### 2.1 프롬프트 엔지니어링

- 프롬프트 구성은 작업 성능과 모델 해석 가능성 모두를 제어할 수 있다
    - 모델 해석 가능성이란?
        
```text
    💡
    
    ## "모델 해석 가능성(Model Interpretability)"의 의미
    
    ### 정의
    
    모델이 **왜** 그런 답을 내놓았는지 사용자가 **이해할 수 있는 정도**
    
    ### 프롬프트가 해석 가능성을 제어하는 예시
    
    **같은 질문, 다른 프롬프트:**
    
    `질문: "이 회사에 투자해야 할까?"`
    
    | 프롬프트 방식 | 모델 출력 | 해석 가능성 |
    | --- | --- | --- |
    | 그냥 질문 | "투자하지 마세요." | 낮음 (이유 불명) |
    | "이유와 함께 답해줘" | "재무제표 분석 결과... 따라서 투자하지 마세요." | 중간 |
    | "단계별로 추론 과정을 보여줘" | "1단계: 매출 분석... 2단계: 부채 비율... 3단계: 결론..." | 높음 |
    
    ### 핵심 포인트
    
    프롬프트를 **어떻게 작성하느냐**에 따라:
    
    - 모델의 **내부 추론 과정**이 드러날 수도, 숨겨질 수도 있음
    - 사용자가 결과를 **신뢰하고 검증**할 수 있는지가 결정됨
```
        
- 다양한 프롬프트 엔지니어링 기법이 발전하고 있음
    - 형식적인 문법보다는 여러 언어적 표현을 사용/실험해보면서 시행착오에 이존하는 절차적인 특성을 계속 유지하고 있음

#### 의존적인 프롬프트 엔지니어링

- 추론 과정, 생성된 답변의 출력 구조화, 맥락적 의도와 표현방식에 의존함
- 현재 연구들은 의미론적으로 동등한 표현의 사소한 어휘 변형이나 구문 변형에 대한 민감도를 실험/연구함
    - 같은 질문을 했을 때 LLM이 다른 추론, 다른 답변을 할 수있음
    
    → 재현성, 투명성 부족함 
    
    ex) 
    
    - 의미론적으로 동등한 표현 하에서 출력이 변화함
    - 컨텍스트 순서만 변경됐을 때 LLM의 답변이 달라짐
    - 프롬프트를 다듬을 수록 원래 의도와 더 멀어지는 현상

 ⇒ 프롬프트 연구들 결국 3가지 패러다임으로 수렴됨 

#### 정형화된 프롬프트 엔지니어링 연구 패러다임

1. 지시적 프롬프팅 
    - 작업 목표화 스타일, 제약 조건을 자연어로 직접 입력함
        
        ex) 공식적인 어조로 200 단어 요약문을 작성해줘
        
    - 직관적지만 장황함, 일관성 없음 → 패러프레이즈 전반에 모호함
2. 추론 프롬프팅(CoT), ReAct
    - 해석 가능성을 향상시켜줌
    - 어떤 추론 행동이 강제되어야 하는지가 아닌 추론이 어떻게 진행되는지에 대해서만 알 수 있음
3. 메타 프롬프팅 
    - 프롬프트를 모듈화해서 재사용 가능함
    - 매개변수화, 체이닝, 버전 관리를 도입하지만 비개발자의 접근성이 제한됨

### 2.2 구조화된 프롬프팅 프레임워크

> 비구조화된 자연어 프롬프트의 취약성에 대응해서 최근 연구에서는 자연어 대신 정해진 문법으로 AI를 제어하는 프레임워크로 발전하고 있음
> 

**LMQL (Language Model Query Language)** 

- 변수, 제약, 제어 흐름을 통합된 구문으로 통합하여 프롬프팅을 쿼리 실행 프로세스로 재구성
    - 쿼리 실행 프로세스로 재구성한다?
        
```text
    💡
    
    ### 비유: 식당 주문
    
    **일반 프롬프트 (자연어):**
    
    ```
    "음... 맵지 않고, 고기 들어가고, 가격은 만원 이하인 거 주세요. 
    아, 근데 너무 짜지 않게요."
    ```
    
    → 점원이 알아서 해석 (결과 불확실)
    
    **LMQL 방식:**
    
    ```
    주문:
      - 맵기: 0단계
      - 재료: 고기 포함
      - 가격: < 10000원
      - 염도: 낮음
      
    → 조건에 맞는 메뉴만 출력
    ```
    
    → 조건을 **정확히** 걸어서 결과 통제
    
```
        
- 자연어 세그먼트를 실행 가능한 쿼리로 컴파일 함으로써 추론 비용을 26~85%까지 줄임

→ 구조화된 제어가 효율성과 결정론을 모두 증가시킬 수 있음 

**BAML (Behavioral API Modeling Language)**

- 모델 상호작용을 타입화된 행동 API로 형식화한다
    - 예시

    ```text
    💡 일반 프롬프트 (자유 형식):

    "네 정보 알려줘"
    → AI 응답: "저는 홍길동이고 25살이에요 서울 살아요"
      (형식 뒤죽박죽, 파싱 어려움)

    BAML 방식 (양식 지정):

    [입력 양식]
    질문: string

    [출력 양식]
    이름: string
    나이: integer
    주소: string

    → AI 응답:
    {
      "이름": "홍길동",
      "나이": 25,
      "주소": "서울"
    }
    → 반드시 이 형식으로만 나옴
    ```
        
- 입/출력 형식을 명시적 스키마를 정의해서 재현성을 향상시킴

**DSPy (Declarative Structured Programming for LLMs)**

- 피드백과 학습 신호를 통해 최적화되는 미분 가능한 프롬프트 모듈
- 프롬프트 엔지니어링 자동화 프레임워크

이런 프레임워크들은 LLM 시스템의 제어 및 재현성을 발전시키지만 주로 개발자 친화적임 

→ 사용자 보다는 워크플로우 및 파이프라인 관리를 최적화

### 2.3 LLM 선언적 제어 시스템

*“어떻게 하라고 설명하지 않고, 원하는 결과만 선언하는 방식”*

- 선언적 시스템은 저라적 설명(어떻게 행동하라고 설명) → 행동 명세(무엇을 달성할 것인가)로 이동
- 장황하게 설명하지 않고도 추론 투명성, 톤 조절, 형식 일관성과 같은 행동 요구 사항을 표현
- 데코레이터 프롬프트는 행동 논리를 명시적으로 인코딩하는 사람이 읽음과 동시에 기계가 해석할 수 있는 구문을 정의

## 3. Prompt Decorators 프레임워크

> 논의할 내용이 아니라 어떻게 생각하거나 응답할지를 지정하는 모듈식 지시어 역할을 한다
> 

### 3.1 핵심 개념

- 프롬프트 데코레이터는 사용자 프롬프트를 감싸 모델의 행동을 명시적이고 재현 가능하며 검사 가능하게 만드는 행동 수정자 역할을 한다
- 파이썬 데코레이터에서 아이디어를 얻음

예시 :

```
+++Rewrite
+++Reasoning

Explain the implications of using generative AI in education.
(교육에서 생성형 AI 사용의 함의를 설명하시오)
```

`+++Rewrite` : 모델에게 응답을 생성하기 전에 명확성을 위해 주어진 프롬프트를 재해석하고 다듬도록 지시

`+++Reasoning` : 최종 답변 전에 논리적 설명을 생성하도록 요구함 

→ 선언적 구성이 기본 지시사항을 변경하지 않고 모델 행동의 인지적 순서를 지정해줌 

### 3.2 Prompt Decorators 구문 및 범위

**기본 형식:**

`+++Name(parameter=value)`

| 구성 요소 | 설명 |
| --- | --- |
| `+++` | 데코레이터 시작 표시 |
| `Name` | 데코레이터 이름 |
| `(parameter=value)` | 선택적 매개변수 |

**특징:**

- 여러 개 쌓기 가능
- 위에서 아래로 순서대로 처리

### 3.3 Prompt Decorators Processing

1. **Parsing** : 데코레이터 이름, 매개변수 및 순서를 추출
2. **Scope Resolution** : 채팅 범위와 메시지 범위를 병합하고, 지우기 규칙 적용 
    
    : 어떤 데코레이터가 지금 이 메시지에 적용되는지 결정
    
3. **Planning & Interaction** : 계획 수립, 명확화 질문, 개념적 맥락 가져오기
4. **Reasoning & Generation** : 구조화된 분석, 토론, 비평, 개선 수행
5. **Formatting & Expression** : 어조, 스타일, 출력 형식 제약 적용
6. **Introspection & Export** : 현재 상태 공개 또는 감사용 메타데이터 내보내기
    
    : 현재 설정 상태를 보여주고, 대화 기록을 저장

### 기능적 범주 및 하위 범주별 프롬프트 데코레이터 분류 체계

| 범주 | 하위 범주 | 데코레이터 | 기능/설명 |
| --- | --- | --- | --- |
| **인지적 & 생성적** | 추론 & 생성 | `+++Reasoning` | 투명성과 추적 가능성을 위해 최종 답변 전에 추론 제공 |
|  |  | `+++StepByStep` | 명확한 진행을 위해 레이블된 단계로 작업 실행 |
|  |  | `+++Debate` | 종합 전에 여러 입장 제시 |
|  | 질의 & 명확화 | `+++Interactive` | 프롬프트가 모호할 때 명확화 질문 요청 |
|  |  | `+++Socratic` | 가정을 드러내고 이해를 심화하기 위해 소크라테스식 질문 적용 |
|  | 계획 & 아이디어 | `+++Planning` | 작업 실행 전에 계획과 목표 개요 작성 |
|  |  | `+++Brainstorm` | 즉각적인 평가 없이 여러 레이블된 아이디어 생성 |
|  |  | `+++Rewrite` | 사용자 프롬프트를 더 명확한 버전으로 재구성 |
|  |  | `+++Import` | 개념적 렌즈나 학문적 관점 도입 |
|  | 평가 & 피드백 | `+++Critique` | 강점, 약점, 개선점이 포함된 구조화된 피드백 제공 |
|  |  | `+++Refine` | 세련화 주기를 통해 출력을 반복적으로 개선 |
|  |  | `+++Candor` | 피드백의 직접성과 솔직함 수준 제어 |
| **표현적 & 시스템적** | 출력 형식 | `+++OutputFormat` | 구문적으로 유효한 출력 형식 강제 (JSON, YAML, Markdown 등) |
|  |  | `+++Tone` | 어조나 스타일 레지스터 구성 (공식적, 친근한 등) |
|  | 세션 & 메타 제어 | `+++ChatScope` | 채팅 턴 간에 지속적인 행동 활성화 |
|  |  | `+++MessageScope` | 데코레이터 효과를 현재 메시지로 제한 |
|  |  | `+++Clear` | 채팅 범위에서 전체 또는 선택된 데코레이터 제거 |
|  |  | `+++ActiveDecs` | 현재 채팅의 모든 활성 데코레이터 나열 |
|  |  | `+++AvailableDecs` | 지원되는 데코레이터 카탈로그와 상태 표시 |
|  |  | `+++Export` | 기록 보관을 위해 대화 내용과 메타데이터 내보내기 |

### 각 데코레이터 상세 설명

#### 인지적 & 생성적 데코레이터

| 데코레이터 | 설명 | 출력 형식 | 사용 상황 |
| --- | --- | --- | --- |
| `+++Reasoning` | "왜 그런 결론을 내렸는지" 생각 과정을 먼저 보여준 후 답변 제공 | `[Reasoning]` → `[Final Answer]` | "이 주식 사야 할까?" 같은 판단이 필요한 질문 |
| `+++StepByStep` | 답을 한 번에 주지 않고, 1단계→2단계→3단계 순서대로 나눠서 설명 | `Step 1` → `Step 2` → `Final Step` | 수학 문제 풀이, 코드 작성 과정, 요리 레시피 |
| `+++Debate` | 한쪽 의견만 주지 않고, 찬성/반대 양쪽 입장을 모두 보여준 후 종합 | `Position A` → `Position B` → `Synthesis` | "재택근무가 좋을까?" 같은 찬반이 있는 주제 |
| `+++Interactive` | 질문이 모호하면 바로 답하지 않고, "이게 맞나요?" 되물어서 확인 후 진행 | 질문 → 답변 수신 → 작업 계속 | "그거 정리해줘" 같은 애매한 요청 |
| `+++Socratic` | 답을 바로 알려주지 않고, 질문을 던져서 스스로 생각하게 유도 | 재진술 → 가정 탐색 → 종합 | 학습 목적, 개념 이해 심화 |
| `+++Planning` | 바로 작업 시작하지 않고, "이렇게 진행하겠습니다" 계획을 먼저 세움 | `[Plan]` → `[Execution]` | 보고서 작성, 프로젝트 기획, 복잡한 작업 |
| `+++Brainstorm` | 좋고 나쁨 평가 없이 일단 아이디어를 여러 개 쏟아냄 | `Option 1`, `Option 2`, `Option 3`... | "마케팅 아이디어 뭐가 있을까?" 같은 발산적 질문 |
| `+++Rewrite` | 사용자의 애매한 질문을 더 명확하게 다시 써준 후, 그 질문에 답변 | `[Rewritten Prompt]` → `[Response]` | "AI 좋아?" → "AI의 장단점은 무엇인가요?"로 개선 |
| `+++Import` | 특정 분야의 관점으로 분석 (예: 심리학적 관점, 경제학적 관점) | 가져온 개념 식별 → 일관되게 적용 | "이 문제를 시스템 사고로 분석해줘" |
| `+++Critique` | 칭찬만 하거나 비판만 하지 않고, 강점/약점/개선점을 구조적으로 피드백 | 강점 → 약점 → 개선 → 결론 | 이력서 검토, 사업계획서 피드백, 글 첨삭 |
| `+++Refine` | 한 번에 완성하지 않고, 1차→2차→3차 반복하며 점점 다듬기 | `Iteration 1` → `Iteration 2` → `Final` | 중요한 이메일 작성, 발표 자료 완성도 높이기 |
| `+++Candor` | 피드백의 솔직함 정도 조절 (부드럽게 vs 직설적으로) | `low`: 돌려말함 / `medium`: 적당히 / `high`: 직설적 | 상사에게 vs 친한 동료에게 피드백할 때 |

#### 표현적 & 시스템적 데코레이터

| 데코레이터 | 설명 | 매개변수/옵션 | 사용 상황 |
| --- | --- | --- | --- |
| `+++OutputFormat` | 출력 형식을 강제로 지정 (JSON, 표, 마크다운 등) | `format=` JSON, YAML, Markdown, XML | 코드에서 바로 쓸 데이터, API 연동 |
| `+++Tone` | 말투/분위기 설정 (격식체, 친근체, 전문용어 등) | `style=` formal, casual, technical, friendly | 공식 문서 vs 친구에게 설명 |
| `+++ChatScope` | 이 설정을 이번 메시지만 아니라 대화 끝까지 유지 | 없음 | "앞으로 계속 존댓말로 해줘" |
| `+++MessageScope` | ChatScope 무시하고, 이번 메시지만 다른 설정 적용 | 없음 | "이번만 반말로 해줘" |
| `+++Clear` | 지금까지 설정한 데코레이터 초기화 (전체 또는 선택) | 없음: 전체 리셋 / `(+++Name)`: 특정 것만 | "설정 초기화하고 새로 시작" |
| `+++ActiveDecs` | 지금 어떤 데코레이터가 켜져있는지 목록 확인 | 없음 | "지금 무슨 설정 적용 중이지?" |
| `+++AvailableDecs` | 사용할 수 있는 전체 데코레이터 목록 보기 | 없음 | "어떤 데코레이터가 있지?" |
| `+++Export` / `+++Dump` | 지금까지의 대화 내용과 설정을 파일로 저장/내보내기 | `format=` 텍스트, Markdown, JSON | 대화 기록 보관, 나중에 다시 보기 |

### 5. 프롬프트 데코레이터의 응용 사례

#### 5.1 다관점 기능 평가

**데코레이터:** `+++Debate` + `+++Reasoning` + `+++Refine(iterations=3)`

| 항목 | 내용 |
| --- | --- |
| 문제점 | 새 기능 평가 시 단일 이해관계자 관점으로 편향 |
| 해결책 | Debate로 찬반 생성 → Reasoning으로 논리화 → Refine으로 종합 |
| 결과 | 여러 분야 트레이드오프를 포착한 추적 가능한 의사결정 근거 |

**예시 상황:** "AI 요약 기능을 베타 출시해야 할까?"

#### 5.2 가설 구조화 도우미

**데코레이터:** `+++Socratic` + `+++Planning` + `+++StepByStep`

| 항목 | 내용 |
| --- | --- |
| 문제점 | 연구자의 모호한 아이디어 → 논리적으로 일관되지 않은 가설 |
| 해결책 | Socratic으로 가정 표면화 → Planning으로 구조화 → StepByStep으로 순차 논리화 |
| 결과 | 개념에서 검증 가능한 가설까지 명시적 추론 단계 제공 |

**예시 상황:** "피드백 빈도가 팀 창의성에 영향을 미칠 것 같아" → 검증 가능한 가설로 구체화

#### 5.3 건설적 피드백 작성기

**데코레이터:** `+++Candor` + `+++Tone` + `+++Refine(iterations=2)`

| 항목 | 내용 |
| --- | --- |
| 문제점 | 너무 직설적 → 신뢰 손상 / 너무 모호 → 성과 명확성 저해 |
| 해결책 | Candor로 솔직함 보장 → Tone으로 전문적 공감 → Refine으로 표현 조정 |
| 결과 | 진정성 유지하면서 대인관계 마찰 최소화 |

**예시 상황:** "실력은 좋은데 마감을 자주 놓치는 팀원에게 피드백 작성"

#### 5.4 연구 제목 아이디어 도우미

**데코레이터:** `+++Brainstorm` + `+++Reasoning` + `+++Refine(iterations=3)`

| 항목 | 내용 |
| --- | --- |
| 문제점 | AI 제안이 일반적이거나 학문적 관례에 맞지 않음 |
| 해결책 | Brainstorm으로 후보 다량 생성 → Reasoning으로 근거 설명 → Refine으로 필터링 |
| 결과 | 분야에 적합하고 잘 추론된 제목 |

**예시 상황:** "인간-AI 협업 정책 분석 논문 제목 추천해줘"

#### 5.5 프롬프트 개선 도우미

**데코레이터:** `+++Rewrite` + `+++Reasoning`

| 항목 | 내용 |
| --- | --- |
| 문제점 | 사용자의 모호한 프롬프트 → 불충분한 출력 |
| 해결책 | Rewrite로 프롬프트 명확화 → Reasoning으로 투명한 추론 |
| 결과 | 프롬프트 품질 향상 + 사용자 프롬프트 리터러시 학습 |

**예시 상황:** "광합성 수업용으로 설명해줘" → "중학생 대상, 햇빛·엽록소·에너지 변환 강조" 로 개선

#### 5.6 사례 간 통찰과 설계 원칙

**공통 패턴: 인지적 제어 + 표현/개선 데코레이터 조합**

| 데코레이터 조합 | 추론 원형 |
| --- | --- |
| `Debate` + `Reasoning` + `Refine` | 변증법적 숙의 → 균형 잡힌 결정 |
| `Socratic` + `Planning` + `StepByStep` | 탐구 주도 → 논리적 가설 구조화 |
| `Candor` + `Tone` + `Refine` | 감정 지능적 의사소통 → 어조 조정 |
| `Brainstorm` + `Reasoning` + `Refine` | 발산 → 분석적 필터링 → 수렴 |
| `Rewrite` + `Reasoning` | 프롬프트 개선 → 투명한 추론 |

## 6. Risks and Limitations

**1. 가짜 투명성 주의**

- `+++Reasoning`은 "진짜 생각 과정"이 아님
- 그럴듯해 보이는 설명일 뿐

**2. 조합 시 충돌**

- 여러 데코레이터 함께 쓰면 서로 간섭 가능
- 형식이 추론 자르거나, 개선이 어조 덮어씀

**3. 보안 취약점**

- 사용자가 `+++Clear` 몰래 삽입 → 안전장치 해제 가능
- 프롬프트 인젝션 공격에 노출

## 시스템 프롬프트

- 결국 시스템 프롬프트가 늘어나는 구조

prompt-decorators.txt

- 요약
    
    문제: LLM에게 추론 깊이, 어조, 출력 형식을 제어하려면 매번 장황하고 일관되지 않은 자연어 지시를 써야 함 → 재현성 없고, 결과 예측 불가
    
    해결책: 자연어 지시 대신 표준화된 기호(`+++Decorator`)로 행동을 선언하는 방식
    
    사용 예시:
    
    기존 방식:
    "단계별로 생각하고, 공식적인 어조로, 마크다운 형식으로
    찬반 양론을 모두 다뤄서 분석해줘..."
    
    데코레이터 방식:
    +++Reasoning
    +++Debate
    +++Tone(style=formal)
    +++OutputFormat(format=markdown)
    
    AI 채용 시스템의 윤리적 함의를 평가하시오.
    
    결과:
    
    - 프롬프트 복잡성 감소 (장황한 지시 → 간결한 기호)
    - 세션 간 재현 가능한 행동
    - 감사 가능한 투명한 설정 (어떤 데코레이터 적용 중인지 확인 가능)
    - 6가지 응용 사례에서 추론 투명성, 작업 신뢰성 향상 확인
    
    한계:
    
    - 아직 실험적인 단계인것같음
    - 같은 데코레이터도 세션/모델마다 다르게 해석될 수 있음
    - `+++Reasoning`이 보여주는 건 "진짜 사고 과정"이 아닌 시뮬레이션
    - 여러 데코레이터 조합 시 충돌 가능 (형식이 추론 자름 등)
    - 프롬프트 인젝션 공격에 취약 (가짜 `+++Clear` 삽입 등)
